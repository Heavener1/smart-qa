### 智能问答系统项目报告书

#### 一、项目概述

**1.1 项目背景**
随着企业数字化转型的深入，内部知识资产（如技术文档、产品手册、规章制度等）呈爆炸式增长。员工在查找信息时面临“信息孤岛”和“检索效率低”的难题。同时，客户服务场景对响应速度和准确性提出了更高要求。传统的关键词检索已无法满足需求，基于大语言模型（LLM）和检索增强生成（RAG）技术的智能问答系统成为解决这一痛点的关键。

**1.2 项目目标**
本项目旨在构建一个功能全面、高度可配置、安全可靠的智能问答系统。该系统将整合企业内部多源异构数据，通过灵活的“场景”配置，为不同业务线（如IT支持、HR咨询、客户服务）提供精准、实时的智能问答服务，从而提升信息获取效率，降低运营成本。

**1.3 项目核心特性**

- **多模态知识库**：支持PDF、Word、PPT、图片、文本等多种格式文档的统一存储与智能解析。
- **灵活的场景配置**：可针对不同应用（如“新员工入职助手”、“产品技术专家”）独立配置问答策略。
- **企业级安全与合规**：内置敏感词过滤、操作日志审计，确保数据安全与合规性。
- **开放的生态集成**：支持接入主流大模型厂商API及自定义插件，扩展系统能力边界。

#### 二、系统核心模块设计

**2.1 知识库管理模块**
知识库是智能问答系统的基石，负责非结构化数据的 ingestion（摄入）、processing（处理）和 storage（存储）。

**2.1.1 多格式文档处理**

- **支持格式**：
- **文本类**：PDF, Word (.docx), PowerPoint (.pptx), Excel (.xlsx), Markdown (.md), TXT。
- **图片类**：JPG, PNG等。通过集成OCR（光学字符识别）服务，提取图片中的文字信息。
- **处理流程**：

1. **文档上传**：用户通过Web界面上传文件。
2. **格式转换与清洗**：

- 使用 `PyPDF2`、`pdfplumber` 等库解析PDF，保留章节结构。
- 使用 `python-docx` 解析Word，将表格内容转换为Markdown格式以便LLM理解。
- 对扫描件PDF和图片，调用OCR服务进行文字识别。

3. **智能文本切分（Chunking）** ：

- 采用 `RecursiveCharacterTextSplitter` 等策略，按语义完整性将长文档切分为固定长度（如1000字符）且有重叠（如200字符）的文本块（Chunks），以平衡检索精度和上下文完整性。

**2.1.2 基于S3规范的对象存储**

- **存储选型**：采用兼容S3 API协议的开源对象存储系统（如MinIO），实现海量非结构化数据的低成本、高可靠存储。
- **存储结构**：
- **原始文件桶（Raw Documents Bucket）** ：存储用户上传的原始文件。
- **处理后文件桶（Processed Chunks Bucket）** ：存储经过清洗和切分后的文本块（JSON格式），并附带元数据（如 `source_file`, `page_number`, `chunk_id`）。

**2.2 场景配置模块**
“场景”是本系统的核心概念，它将问答流程中的各个组件串联起来，形成一个可复用的问答模板。

**2.2.1 场景配置项**
每个场景都是一个独立的问答应用，其配置包含：

- **场景名称与描述**：如“IT Helpdesk Assistant”。
- **关联知识库**：可选择一至多个已创建的知识库。例如，“IT助手”场景可以关联“公司网络规范”、“常用软件手册”等多个知识库。
- **大模型配置**：指定该场景使用的LLM，并配置其参数。
- **重排序（Re-ranking）条件**：
- **功能**：在向量检索返回初步结果后，使用更精细的模型（如Cohere Rerank）对结果进行二次排序，提升Top-K结果的相关性。
- **配置**：启用/禁用、选择重排序模型、设置返回的最终文档数量（如 `top_k=5`）。
- **关联FAQ库**：选择一个FAQ库，系统会优先在FAQ中进行匹配，若匹配成功则直接返回标准答案，提升响应速度和准确性。
- **应用插件**：选择该场景可以调用的插件列表，如“查询天气”、“创建JIRA工单”等。
- **关联敏感词库**：选择一个敏感词库，对用户提问和模型回答进行双重过滤。
- **系统提示词（System Prompt）** ：为场景定制角色和回答风格。例如：“你是一个专业的IT支持专家，请用简洁的语言回答用户问题，如果知识库中没有相关信息，请回答‘抱歉，我暂时无法在现有资料中找到答案’。”

**2.2.2 场景管理**

- 支持创建、编辑、复制、删除场景。
- 支持场景的启用/禁用。

**2.3 大模型配置模块**
为满足不同成本、性能和数据安全的需求，系统支持接入多个厂商的大模型。

**2.3.1 支持的模型类型**

- **云端API模型**：OpenAI (GPT-4o), Anthropic (Claude 3.5 Sonnet), Google (Gemini 1.5 Pro)等。
- **本地/私有化部署模型**：通过Ollama、vLLM等服务部署的开源模型，如Qwen2.5-7B-Instruct, Llama 3等。
- **模型配置参数**：
- **模型ID/名称**：如 `gpt-4o`, `qwen2.5-7b-instruct`。
- **API端点（API Base URL）** ：模型服务的访问地址。
- **API密钥（API Key）** ：访问凭证。
- **上下文窗口（Context Window）** ：模型能处理的最大token数。
- **推理参数**：
- **温度（Temperature）** ：控制回答的随机性（0-1），知识问答建议设为0.3以保证确定性。
- **最大生成长度（Max Tokens）** ：限制回答长度。

**2.4 FAQ与敏感词配置模块**
**2.4.1 FAQ配置**

- **功能**：管理“问题-答案”对，用于处理高频、标准的问题。
- **配置项**：
- **问题（Question）** ：支持添加多个相似问法。
- **答案（Answer）** ：标准回复内容。
- **相似度阈值**：设置用户问题与FAQ问题匹配的最低相似度分数。

**2.4.2 敏感词配置**

- **功能**：构建敏感词库，保障内容安全。
- **配置项**：
- **敏感词列表**：支持批量导入和手动添加。
- **过滤动作**：
- **用户输入过滤**：当用户问题命中敏感词时，系统可直接拒绝回答或进行替换。
- **模型输出过滤**：对模型生成的回答进行后置审查，若包含敏感词则进行屏蔽或替换。

**2.5 插件接入模块**
插件系统用于扩展问答机器人的能力，使其能与外部系统交互。

**2.5.1 插件架构**

- 采用标准化的插件接口，每个插件需实现 `name`, `description`, `parameters`, `execute` 等核心方法。
- 插件以Python模块形式开发，并注册到系统中。

**2.5.2 插件示例**

- **天气查询插件**：调用第三方天气API，返回指定城市的天气信息。
- **工单创建插件**：对接JIRA或内部工单系统，根据用户描述自动创建问题工单。
- **数据库查询插件**：在授权范围内，执行预设的SQL查询，返回结构化数据。

**2.6 日志信息记录模块**
完整的日志系统是系统运维、问题排查和效果优化的基础。

**2.6.1 日志类型**

- **用户会话日志**：记录完整的用户提问、系统检索到的上下文、模型生成的最终回答、耗时等信息。
- **系统操作日志**：记录知识库上传、场景配置修改、模型参数调整等关键操作。
- **错误日志**：记录系统运行中出现的异常，如API调用失败、文档解析错误等。

**2.6.2 日志存储与分析**

- 所有日志统一写入Elasticsearch或类似搜索引擎中，便于进行全文检索和聚合分析。
- 提供可视化仪表盘，展示问答量、高频问题、用户满意度等关键指标。

#### 三、系统技术架构

系统采用前后端分离的微服务架构。

**3.1 前端（Frontend）**

- **技术栈**：Vue.js / React
- **功能**：提供知识库管理、场景配置、模型配置、FAQ管理、日志查看等Web管理界面，以及面向最终用户的聊天窗口。

**3.2 后端（Backend）**

- **技术栈**：Python (FastAPI)
- **核心框架**：LangChain / LlamaIndex
- **功能**：
- **API服务**：提供RESTful API供前端调用。
- **RAG引擎**：实现文档加载、文本切分、向量化、检索、重排序、提示词构建和LLM调用的核心流程。
- **插件调度器**：根据场景配置，动态加载和执行插件。
- **日志记录器**：负责各类日志的异步写入。

**3.3 数据存储层（Data Layer）**

- **对象存储（S3-compatible）** ：MinIO，用于存储原始文档和处理后的文本块。
- **向量数据库（Vector Database）** ：Pinecone / Weaviate / Chroma，用于存储文本块的向量嵌入（Embedding），并提供高效的相似性检索。
- **关系型数据库（Relational Database）** ：PostgreSQL / MySQL，用于存储场景配置、模型配置、FAQ、敏感词、用户信息等结构化数据。
- **日志存储（Log Storage）** ：Elasticsearch，用于存储和检索海量日志数据。

#### 四、系统工作流程

1. **用户提问**：用户在聊天界面输入问题。
2. **输入审查**：系统首先根据场景配置的敏感词库对用户问题进行过滤。
3. **FAQ优先匹配**：系统在关联的FAQ库中进行检索，若找到高匹配度的问题，则直接返回对应答案，流程结束。
4. **知识库检索**：若FAQ未命中，系统在关联的知识库中进行向量检索，获取最相关的文本块（Chunks）。
5. **结果重排序**：如果启用了重排序，则对检索结果进行二次排序，选出相关性最高的Top-K个文本块。
6. **插件调用（可选）** ：根据场景配置和LLM的判断，决定是否调用一个或多个插件来获取额外信息。
7. **构建提示词**：将用户问题、检索到的上下文、插件返回的结果以及系统提示词（System Prompt）组合成一个完整的提示词（Prompt）。
8. **LLM生成回答**：调用场景配置中指定的大模型，传入构建好的提示词，生成最终回答。
9. **输出审查**：对模型生成的回答进行敏感词过滤。
10. **返回结果并记录日志**：将最终回答返回给用户，并将完整的会话信息异步写入日志系统。

#### 五、项目风险与应对策略

表格

| 风险类别 | 风险描述                                     | 应对策略                                                                                            |
| ---------- | ---------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **技术风险**         | 检索结果不准确，导致模型“幻觉”或回答错误。 | 1. 优化文档切分策略。2. 启用并调优重排序模型。3. 在提示词中明确要求模型“仅根据提供的上下文回答”。 |
| **性能风险**         | 随着知识库规模增大，检索和问答延迟增加。     | 1. 使用高性能向量数据库。2. 对高频问题进行答案缓存。3. 对大型文档进行异步处理。                     |
| **安全风险**         | 敏感数据泄露或被恶意利用。                   | 1. 部署在内网环境。2. 严格的敏感词过滤。3. 完善的权限控制和操作审计。                               |
| **成本风险**         | 调用商业大模型API成本高昂。                  | 1. 对于非核心场景，优先使用本地部署的开源模型。2. 优化提示词，减少不必要的token消耗。               |

#### 六、总结与展望

本项目设计并实现了一个功能强大、高度灵活的智能问答系统。通过“场景”这一核心抽象，系统能够轻松适配企业内部多样化的问答需求。未来，我们将持续探索以下方向：

- **多模态问答**：支持直接上传截图或图表进行提问。
- **智能体（Agent）化**：让系统具备更复杂的任务规划和执行能力。
- **效果自优化**：基于用户反馈和日志数据，自动优化检索和重排序策略。
